# Python 主题维护规则

## 这页是干什么的

这页不是知识内容本身，而是“怎么维护这本小书”的说明。

目的很明确：
- 让内容越积越顺
- 尽量避免越记越乱
- 保证后续聊天成果能稳定补到正确位置
- 方便未来迁移到博客

## 当前维护原则

### 1. 优先补已有文档，不优先新建文件
后续如果继续聊到：
- Python 基础
- Web 后端概念
- [`yy-auth`](../../yy-auth) 项目理解
- 名词解释
- 命令记录

优先做法是：
- 找对应文档
- 补到对应章节
- 只有当主题明显独立，才新建文件

目标是让这一部分越来越像一本书，而不是一个文件越堆越多的仓库。

### 2. 聊天成果要尽量沉淀成正式内容
如果后续聊天里出现这些内容：
- 一个明确的认知结论
- 一个清晰的项目理解
- 一个高频命令总结
- 一个术语的通俗解释
- 一个容易忘的坑

就应该尽量沉淀到正式文档里，而不是只停留在聊天记录里。

### 3. 临时笔记和正式内容分开
不是所有内容都适合直接写进正式正文。

临时内容可以先进入笔记区，后面再整理。

后续推荐的区分方式：
- 正式内容：放在主题文档里
- 临时笔记：放在 notes 目录里
- 外部资料摘录：放在单独的 external notes 文档里

### 4. 尽量用“自己的话”沉淀
知识库不是复制粘贴仓库。

后续整理时优先保留这些特点：
- 讲人话
- 能复述
- 能回头看懂
- 必要时允许用一些比较“俗”的表达帮助记忆

目标不是写得像教材，而是写得像未来的自己还能看懂、还能继续接着学。

### 5. 术语尽量统一
一个词尽量保持：
- 称呼稳定
- 解释稳定
- 读法稳定

优先沉淀到 [`glossary.md`](./glossary.md)，避免同一个词在不同文章里越说越乱。

### 6. 项目实战内容要尽量关联真实仓库
像 [`yy-auth`](../../yy-auth) 这类内容，不要只写抽象结论。
尽量保留：
- 这是哪个项目
- 在哪里看到的
- 它在项目里起什么作用
- 为什么这个点值得记

这样后面回头看时，不至于只剩空话。

## 后续我和 AI 的协作方式

后面如果要让我补知识库，你可以直接用这种说法：
- 把今天聊的内容补到 Python 知识库里
- 把关于 [`yy-auth`](../../yy-auth) 的整体认知整理到对应位置
- 把这些术语补到 [`glossary.md`](./glossary.md)
- 把这段外部资料先放到笔记区
- 把这段内容整理得更像正式文档
- 给这部分补一张 Mermaid 图

我默认会做的事：
1. 先判断该放哪
2. 优先补到已有文档
3. 只在必要时新增文件
4. 尽量保持整体结构稳定

## 我自己补充内容时的建议

如果你自己改文档，建议尽量做这几件事：
- 标清楚是正式理解还是临时备注
- 如果是外部资料，尽量记一下来源
- 如果是暂时没想明白的问题，可以写成“待验证问题”
- 能不用碎片化随手堆，就尽量别堆

## 面向未来博客迁移的约束

为了以后迁移到博客更顺，当前文档尽量遵守：
- 用标准 Markdown
- 文件名语义清晰
- 一级标题只保留一个
- 层级不要太乱
- 图尽量用 Mermaid 或标准图片引用
- 内容尽量能一篇篇单独成文

## 当前一句话原则

**先沉淀到对的位置，再慢慢长厚；不要为了“记了很多”而把知识库重新搞乱。**
