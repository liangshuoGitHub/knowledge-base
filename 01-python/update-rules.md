# Python 主题维护规则

## 这页是干什么的

这页不是 Python 知识内容本身，也不是给 AI 的底层执行约束。

它更偏向：
- 让你以后自己回看时，知道这个主题应该怎么继续长
- 让这个主题保持稳定的写作方向
- 避免内容越来越多以后重新变乱

如果是“提交信息必须中文”“哪些文件名不要乱加链接”这类更偏执行层的规则，已经收敛到知识库仓库内的 `.claude/knowledge-base-maintenance/SKILL.md` 里。

这页保留的是：**从人的视角看，Python 主题应该怎么维护。**

## 这个主题后续怎么长

### 1. 优先补到已有文档，不急着新建页面
后续如果继续聊到这些内容：
- Python 基础
- Web 后端概念
- [`yy-auth`](../../yy-auth) 项目理解
- 名词解释
- 常用命令

优先做法是：
- 先找对应文档
- 补到对应章节
- 只有当主题明显独立时，才新建文件

目标是让 Python 这部分越来越像一本小书，而不是页面越长越散。

### 2. 聊天成果尽量沉淀成正式内容
如果后续聊天里已经出现：
- 一个明确的认知结论
- 一个高频命令总结
- 一个术语的通俗解释
- 一个真实项目里的关键理解
- 一个以后大概率还会忘的坑

就尽量把它沉淀进正式文档，而不是只留在聊天记录里。

### 3. 尽量写成“自己的话”
知识库不是资料搬运仓库。

后续整理时优先保留这些特点：
- 讲人话
- 能复述
- 能回头看懂
- 必要时可以用一些更接地气的表达帮助记忆

目标不是写得像教材，而是写得像未来的自己还能继续接着学。

### 4. 术语尽量统一
同一个词尽量保持：
- 称呼稳定
- 解释稳定
- 读法稳定

优先沉淀到 [`glossary.md`](./glossary.md)，避免同一个词在不同文章里越说越乱。

### 5. 项目实战内容尽量关联真实仓库
像 [`yy-auth`](../../yy-auth) 这类内容，不要只写抽象结论。

尽量保留这些信息：
- 这是哪个项目
- 在哪里看到的
- 它在项目里起什么作用
- 为什么这个点值得记

这样后面回头看时，不至于只剩空话。

## 你以后可以怎么继续补

如果后面继续维护 Python 主题，可以直接用这种说法：
- 把今天聊的内容补到 Python 知识库里
- 把关于 [`yy-auth`](../../yy-auth) 的整体认知整理到对应位置
- 把这些术语补到 [`glossary.md`](./glossary.md)
- 把这段内容整理得更像正式文档
- 给这部分补一张 Mermaid 图

## 你自己补内容时，最值得记住的几件事

如果你自己改文档，尽量注意：
- 优先补到现有页面，不随手横向开新坑
- 如果是外部资料，尽量记一下来源
- 如果是暂时没想明白的问题，可以写成“待验证问题”
- 能整理成正式结论时，就尽量别只留碎片

## 面向未来博客迁移时，继续保持这些习惯

为了以后迁移到博客更顺，当前文档尽量继续遵守：
- 用标准 Markdown
- 文件名语义清晰
- 一级标题只保留一个
- 层级不要太乱
- 图尽量用 Mermaid 或标准图片引用
- 内容尽量能一篇篇单独成文

## 当前一句话原则

**先把内容沉淀到对的位置，再慢慢长厚；不要为了“记了很多”，把 Python 主题重新搞乱。**
