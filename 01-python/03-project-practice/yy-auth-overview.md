# yy-auth 项目整体认知

## 这篇是干什么的

这篇的目标不是把 [`yy-auth`](../../../../yy-auth) 所有源码讲完。

它主要解决一个很现实的问题：

**第一次看公司 Python 项目时，先别被一堆文件和陌生词吓住，先建立整体认知。**

也就是说，这篇更像“先看地图”，不是“立刻钻进每一行代码”。

## 先说一句人话版结论

[`yy-auth`](../../../../yy-auth) 可以先理解成：

**一个统一认证 / 用户 / 权限 / 组织管理的后端服务。**

再说得更接地气一点：
它很像很多业务系统背后那个“总账号中心 + 权限中心 + 组织管理后台”的后端。

所以它不是单纯一个登录接口项目，而是更偏平台型服务。

## 它大概在管什么

从目前看到的代码和结构，可以先把它的职责理解成这些：
- 用户登录、注销、当前用户信息
- JWT 鉴权
- 角色和权限管理
- 组织、空间、系统配置管理
- SSO / OAuth2 / 外部供应商用户接入
- 登录统计

这类服务最大的特点就是：
- 名词多
- 配置多
- 依赖多
- 路由多
- 中间件多

所以第一次看很容易脑子发胀，这很正常。

## 技术上它是什么项目

从 [`requirements.txt`](../../../../yy-auth/requirements.txt) 来看，它是一个典型的 Python 企业后端服务，核心栈包括：
- [`FastAPI`](../../../../yy-auth/requirements.txt)
- [`uvicorn`](../../../../yy-auth/requirements.txt)
- [`gunicorn`](../../../../yy-auth/requirements.txt)
- [`pydantic`](../../../../yy-auth/requirements.txt)
- [`SQLAlchemy`](../../../../yy-auth/requirements.txt)
- [`Redis`](../../../../yy-auth/requirements.txt)
- [`Nacos`](../../../../yy-auth/app/core/zn_nacos.py)

如果只说一句：
**它不是脚本项目，而是一个正式企业服务。**

## 我当前应该先怎么理解它

### 第一层：它是个接口服务
入口在 [`app/main.py`](../../../../yy-auth/app/main.py)。

里面能看到：
- 它基于 [`FastAPI`](../../../../yy-auth/app/main.py:21)
- 启动时会注册中间件
- 会初始化数据库和缓存
- 会注册一批路由

也就是说，它首先是一个标准的 Web 后端服务。

### 第二层：它不是孤零零一个服务
它依赖的东西不少，比如：
- [`MySQL`](../04-common-infra/mysql.md)
- [`Redis`](../04-common-infra/redis.md)
- Nacos 配置中心
- 统一鉴权能力
- 可能还有外部 SSO / OAuth2 服务

这说明它处在公司的整体平台体系里，不是单机小玩具。

这里有一个今天很值得记住的认知：
- [`MySQL`](../04-common-infra/mysql.md) 更偏长期结构化数据
- [`Redis`](../04-common-infra/redis.md) 更偏高频临时数据

所以在项目里同时看到它们，不是重复建设，而是分工不同。

### 第三层：它很适合我用前端视角去反推
这是我当前最有优势的地方。

我可以把它映射成自己熟悉的前端场景：
- 登录页背后的接口是谁
- 当前用户信息从哪来
- 菜单权限 / 按钮权限后端是怎么组织的
- 系统配置页后端怎么存
- 路由和权限树是怎么关联的

这样看会比纯后端视角更容易进入状态。

## 当前最值得记住的入口文件

### [`app/main.py`](../../../../yy-auth/app/main.py)
服务入口。

作用：
- 初始化日志
- 拉配置
- 初始化数据库和缓存
- 注册中间件
- 注册路由

可以把它理解成：
**项目开机时的总开关。**

### [`app/routes.py`](../../../../yy-auth/app/routes.py)
总路由分发文件。

作用：
- 把用户、权限、组织、空间、系统管理等模块的接口挂上去

可以把它理解成：
**整个后端项目的总路由表。**

### [`app/middlewares/jwt.py`](../../../../yy-auth/app/middlewares/jwt.py)
JWT 鉴权中间件。

作用：
- 解析 token
- 判断请求是不是登录用户发来的
- 把当前用户信息挂到请求上下文

可以把它理解成：
**后端的门卫之一，先检查你是不是带着合法通行证进门。**

### [`app/middlewares/iam.py`](../../../../yy-auth/app/middlewares/iam.py)
权限校验中间件。

作用：
- 判断当前用户有没有接口访问权限

可以把它理解成：
**第二道门卫，检查你进来了以后有没有资格去干这件事。**

### [`app/apis/user/views.py`](../../../../yy-auth/app/apis/user/views.py)
用户相关接口入口。

作用：
- 登录
- 注销
- 获取当前用户
- 增删改用户
- SSO / OAuth2 等相关能力

可以把它理解成：
**用户体系的大门面。**

### [`app/apis/iam/views.py`](../../../../yy-auth/app/apis/iam/views.py)
权限相关接口入口。

作用：
- 角色管理
- 权限树
- 给角色分配权限
- 判断接口是否有权限

可以把它理解成：
**权限体系的大门面。**

## 这个项目为什么值得作为我的第一站

因为它能一次性让我接触很多真实后端概念，而且这些概念跟前端是有映射关系的：
- 登录态
- token
- 当前用户
- 权限树
- 组织与空间
- 系统配置

对前端转全栈来说，这种项目非常适合当教材。

因为我学到的不是脱离业务的空知识，而是：
- 页面背后的接口
- 接口背后的服务
- 服务背后的鉴权和数据流

## 当前我不该追求什么

现阶段不该追求：
- 一上来把所有代码都看懂
- 把所有 service 文件都啃完
- 一次性记住所有名词
- 一次性理清全部业务

这很容易把自己搞废。

## 当前更适合的做法

现阶段更适合追求：
- 先看懂它是干什么的
- 先知道入口在哪
- 先知道核心路由在哪
- 先知道用户和权限是怎么分开的
- 先跟一条最关键的请求链路

## 当前一句话总结

**[`yy-auth`](../../../../yy-auth) 不是用来一口吃掉的源码山，而是我认识 Python 企业后端项目的第一张地图。**

这句话很重要。

因为只要先有地图，后面再去钻细节，脑子就不会那么乱。
