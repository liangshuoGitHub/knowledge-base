# 知识库使用方法

## 这一部分是干什么的

前面的 Python、Docker 是**知识内容**本身。

这一部分不一样，它装的是「**怎么用好这个知识库**」的方法——也就是元工具：

- 怎么把分散在多个会话里的对话，沉淀成知识库里的正式内容
- 会话结束前怎么收尾，才不会让有价值的东西白白丢掉
- 半成品内容先放哪，避免污染正式文档

简单说：前面是「学到的东西」，这里是「把东西攒下来的方法」。

## 当前目录

- [`session-handoff-template.md`](./session-handoff-template.md)
  - 会话收尾沉淀模板，记录草稿的标准结构。现在入库已自动化，这页主要作为草稿格式参考
- [`inbox.md`](./inbox.md)
  - 统一收件箱，所有会话沉淀的草稿先按时间顺序进这里，整理时再分发进正文

## 为什么需要「多会话沉淀」这套方法

实操时的真实情况是这样的：

- 同一个主题（比如学 FastAPI、搞懂某个真实项目），往往不是一次聊完的
- 单个会话能装的上下文有限，聊着聊着就得开新会话
- 于是一个主题的认知，被切碎在好几个会话里
- 会话一关，里面的结论、踩过的坑、想通的细节，很容易就找不回来了

知识库要解决的就是这个：**把散落在多个会话里的零碎认知，收拢成一篇篇能回看、能复述的正式内容。**

## 标准流程：会话 → 收件箱 → 正式章节

整个沉淀过程分四步，核心是中间加了一个「收件箱」缓冲带，不让半成品直接变正式页：

```
1. 会话产出草稿（自动）
   会话里随时说一句「沉淀知识库草稿」，
   全局 skill knowledge-base-deposit 会基于当前会话上下文，
   自动生成结构化草稿（主题/关联仓库/结论/命令代码/术语/待验证）

2. 草稿自动进统一收件箱
   草稿带写入时间戳，自动追加到 00-index/inbox.md，
   按时间从上到下排列，不急着改写成正式内容

3. 攒够了 / 想清楚了，再提炼成正式章节
   在知识库目录下说一句「整理碎片」，
   仓库 skill knowledge-base-inbox-triage 会把收件箱草稿
   去重、合并、按主题分发，并入正文对应章节

4. 清空已提炼的条目
   已经并入正文的条目，从收件箱删掉，
   保持收件箱只剩还没消化的内容
```

### 为什么要中间这个「收件箱」缓冲带

- **不丢信息**：会话里一句话就能把成果接住，不用当场想清楚怎么归类
- **不污染正文**：半成品、还没想透的点，不会直接塞进正式文档把它搞乱
- **时间线清晰**：所有草稿带时间戳集中在一个文件，整理时一眼看全先后顺序
- **批量提炼质量更高**：攒几次再一起整理，能看出哪些是重复的、哪些能合并成一条主线，比每次单独写更成体系

这和 Python 主题维护规则里那句原则是一致的：**先把内容沉淀到对的位置，再慢慢长厚；不要为了「记了很多」，把主题重新搞乱。**

## 我（使用者）平时只需要说三种话

| 你说 | 在哪说 | 会发生什么 |
|------|--------|-----------|
| 「沉淀知识库草稿」 | 任意会话 | 自动生成结构化草稿，带时间戳追加进 `00-index/inbox.md` |
| 「整理碎片」 | 知识库目录下 | 收件箱草稿去重、合并、分发进正文，然后清空已提炼条目 |
| 「把这几个术语补进 glossary」 | 知识库目录下 | 只动术语表，统一称呼和读法 |

具体由谁生成、放哪、什么时候清空，由两个 skill 约束，你不用每次重复交代：

- 全局沉淀：`~/.claude/skills/knowledge-base-deposit/SKILL.md`
- 仓库整理：`.claude/knowledge-base-inbox-triage/SKILL.md`
- 通用维护规范（表达、链接、中文提交）：`.claude/knowledge-base-maintenance/SKILL.md`
