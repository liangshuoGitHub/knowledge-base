# 知识库使用方法

## 这一部分是干什么的

前面的 Python、Docker 是**知识内容**本身。

这一部分不一样，它装的是「**怎么用好这个知识库**」的方法——也就是元工具：

- 怎么把分散在多个会话里的对话，沉淀成知识库里的正式内容
- 会话结束前怎么收尾，才不会让有价值的东西白白丢掉
- 半成品内容先放哪，避免污染正式文档

简单说：前面是「学到的东西」，这里是「把东西攒下来的方法」。

## 当前目录

- [`session-handoff-template.md`](./session-handoff-template.md)
  - 会话收尾沉淀模板，结束某个会话前复制这段话，让模型自己吐出可直接沉淀的草稿

## 为什么需要「多会话沉淀」这套方法

实操时的真实情况是这样的：

- 同一个主题（比如学 FastAPI、搞懂某个真实项目），往往不是一次聊完的
- 单个会话能装的上下文有限，聊着聊着就得开新会话
- 于是一个主题的认知，被切碎在好几个会话里
- 会话一关，里面的结论、踩过的坑、想通的细节，很容易就找不回来了

知识库要解决的就是这个：**把散落在多个会话里的零碎认知，收拢成一篇篇能回看、能复述的正式内容。**

## 标准流程：会话 → 待整理碎片 → 正式章节

整个沉淀过程分四步，核心是中间加了一个「缓冲带」，不让半成品直接变正式页：

```
1. 会话产出草稿
   会话结束前，用 session-handoff-template.md 里的话，
   让当时的模型吐出结构化草稿（结论/命令代码/术语/待验证/关联仓库）

2. 草稿先进「待整理碎片」区
   把草稿丢给负责维护知识库的模型，先归到对应正式文档末尾的
   「## 待整理碎片」区，不急着改写成正式内容

3. 攒够了 / 想清楚了，再提炼成正式章节
   同一主题的碎片攒了几次、或者某个点已经想明白了，
   就把碎片提炼、去重、重组，并入正文对应章节

4. 清空已提炼的碎片
   已经并入正文的碎片，从「待整理碎片」区删掉，
   保持碎片区只剩还没消化的内容
```

### 为什么要中间这个「待整理碎片」缓冲带

- **不丢信息**：会话一结束就有地方接住，不用当场想清楚怎么归类
- **不污染正文**：半成品、还没想透的点，不会直接塞进正式文档把它搞乱
- **批量提炼质量更高**：攒几次再一起整理，能看出哪些是重复的、哪些能合并成一条主线，比每次单独写更成体系

这和 Python 主题维护规则里那句原则是一致的：**先把内容沉淀到对的位置，再慢慢长厚；不要为了「记了很多」，把主题重新搞乱。**

## 我（使用者）平时只需要说三种话

| 你说 | 会发生什么 |
|------|-----------|
| 「把这段会话内容放进待整理碎片」（附草稿） | 草稿归到对应文档的「## 待整理碎片」区，暂不改写 |
| 「把 XX 主题的碎片提炼成正式章节」 | 碎片去重、合并、重组，并入正文，然后清空已提炼的碎片 |
| 「把这几个术语补进 glossary」 | 只动术语表，统一称呼和读法 |

具体由谁放、放哪、什么时候清空，由知识库维护规则（`.claude/knowledge-base-maintenance/SKILL.md`）约束，你不用每次重复交代。
